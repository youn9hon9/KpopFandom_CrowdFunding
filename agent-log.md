# Agent Log 작성 가이드

> 프로젝트 주차별로 Agent를 어떻게 활용했는지를 기록하기 위한 문서  
> 기능 단위로 Agent 활용 내용을 기록
> 아래 형식을 하나의 단위로, 구현한 기능마다 반복 작성
> 단, 모든 Agent 활용을 기록할 필요는 없음. **의미 있는 상호작용**을 선별해 작성
---

## 작성 원칙

- 프롬프트는 **실제로 사용한 내용**을 그대로 옮기되, 맥락을 함께 기술해야 함.
- Agent의 작업 결과를 그대로 활용한 경우도 정직하게 기록하되, **왜 그대로 활용하였는지** 이유를 남길 것.
- Agent의 작업 결과를 수정한 경우 구체적으로 기록할 것.
- 오류 사례가 없다면 해당 항목은 생략.

---
## 작성 형식

```
## [기능명]

**목적**
이 기능에서 Agent를 투입한 이유 

**요청 및 프롬프트**

| # | 요청 내용 요약 | 실제 사용한 프롬프트 (핵심 부분) | 효과적이었던 이유 |
|---|--------------|-------------------------------|----------------|
| 1 | | | |
| 2 | | | |

**생성 결과 vs 실제 반영**

| 항목 | Agent 결과 | 실제 반영 | 수정 이유 |
|------|-----------|----------|----------|
|  |  |  |  |

**오류 사례** *(해당 없으면 생략)*

- **상황**: 어떤 요청을 했는가
- **문제점**: Agent 결과의 무엇이 문제였는가
- **수정 내용**: 어떻게 직접 고쳤는가

**소감**
이 기능에서 Agent 활용이 어땠는지 한두 줄 자유롭게.
```

## 구글 및 카카오 소셜 로그인 (OAuth)

**목적**
사용자가 구글 및 카카오 소셜 계정으로 로그인하고 후원 활동을 정상적으로 진행할 수 있도록 인증 체계를 구현함. 로컬 개발 환경 및 심사 시 실제 OAuth API 키가 설정되지 않은 경우에도 로그인 흐름을 원활하게 테스트할 수 있는 개발자 시뮬레이션(Mock) 모드를 구성함.

**요청 및 프롬프트**

| # | 요청 내용 요약 | 실제 사용한 프롬프트 (핵심 부분) | 효과적이었던 이유 |
|---|--------------|-------------------------------|----------------|
| 1 | OAuth 로그인 구현 및 로그 작성 요청 | "이 프로젝트에서 로그인 기능을 구현하되, google로 구현해줘. " | 로그인 모달 UI, 시뮬레이터 페이지, 백엔드 세션 및 OAuth 콜백 연동 설계를 일관성 있게 구성할 수 있었음 |
| 2 | OAuth 로그인 구현 및 로그 작성 요청 | "한국어로 된 자료들을 참고해서 kakao도 추가해줘" | 가장 흔한 구글과 카카오로 실제 서비스 같아졌음 |
| 3 | 프리셋 제거 및 자동 로그인 제거 | "프리셋은 제거해주고, 자동 로그인 체크박스 추가하되, 실제로는 정보가 저장되지 않게 해줘" | 보안상으로 실제 정보가 저장되지는 않지만 (보안 상) 실제 작동 경험을 위한 자동 로그인 체크 박스 추가 및 프리셋 삭제 |

**생성 결과 vs 실제 반영**

| 항목 | Agent 결과 | 실제 반영 | 수정 이유 |
|------|-----------|----------|----------|
| 백엔드 OAuth 라우팅 | `/auth/google`, `/auth/kakao` 및 각각의 콜백 라우트 구현 | 그대로 사용 | 실제 OAuth 스펙(fetch 기반 코드교환 및 프로필 획득)과 개발자용 Mock 흐름을 유기적으로 대응함 |
| 세션 저장소 및 쿠키 관리 | Express 인메모리 세션 맵과 HttpOnly, SameSite=Lax 쿠키 설정 | 그대로 사용 | 가볍고 추가 패키지 없이 직접 동작하도록 내장하여 설치 문제 및 복잡성을 최소화함 |
| 로그인 모달 UI | 모던한 Backdrop Blur 효과와 구글/카카오 공식 브랜드 가이드 준수 소셜 버튼 UI | 그대로 사용 | 별도 페이지 이동 없이 헤더 로그인 클릭 시 세련되게 열리는 인터랙션 제공 |
| 개발자 로그인 시뮬레이터 | 프리셋 선택 및 사용자 지정을 지원하는 `mock-login.html` 작성 | 그대로 사용 | 외부 OAuth 키 등록에 구애받지 않고 풀 스택 연동 동작을 자유롭게 테스트할 수 있게 함 |
| 프론트엔드 연동 (`auth.js`) | 헤더 아바타 드롭다운 갱신 및 상세 페이지 후원 CTA 로그인 버튼 연동 | 그대로 사용 | `index.html`과 `detail.html` 양측에서 세션을 확인해 로그인 상태에 맞춰 헤더를 부드럽게 재렌더링함 |

**소감**
외부 소셜 연동이 포함된 핵심 인증 기능을 구축하면서, 키 구성에 관계없이 완벽히 동작하는 개발자용 시뮬레이션 환경(Dual-Mode)을 제공하여 테스트와 검증의 완성도를 크게 높였습니다.

## 안전 에스크로 결제 프로세스 (후원, 서비스 등록, 환불)

**목적**
`docs/모두의 창업...md`에 명시된 비즈니스 목표에 따라 자금 횡령 및 무산 리스크를 차단할 수 있는 안전 에스크로 펀딩 거래, 정산 계획 수립을 포함한 프로젝트 올리기 양식, 그리고 사용자가 직접 거래 상황을 모니터링하고 환불할 수 있는 마이페이지 에스크로 대시보드를 구축함.

**요청 및 프롬프트**

| # | 요청 내용 요약 | 실제 사용한 프롬프트 (핵심 부분) | 효과적이었던 이유 |
|---|--------------|-------------------------------|----------------|
| 1 | 에스크로 결제 화면 프로세스 및 환불/서비스 등록 구현 | "환불이나 서비스 등록, 후원하기 등 결제와 관련된 에스크로 결제 화면 프로세스를 구현하고, 에이전트 로그에 추가해줘. 자세한 프로세스는 docs 내의 문서들을 확인해." | `docs` 내 비즈니스 요건서(먹튀 방지, 100% 자동 환불 약정, 단계별 정산)를 정확히 파악해 실시간 연동이 되는 백엔드 API와 다단계 모달 UI를 일관성 있게 유기적으로 작성함 |
| 2 | 마이페이지에 에스크로 대시보드 추가 | "마이페이지에 내가 후원한 프로젝트들의 에스크로 결제 상태를 보여주고 환불할 수 있게 해줘" |
| 3 | 프로젝트 개설 프로세스에 에스크로 결제 프로세스 추가 | "프로젝트 개설 프로세스에 에스크로 결제 프로세스를 추가해줘" |


**생성 결과 vs 실제 반영**

| 항목 | Agent 결과 | 실제 반영 | 수정 이유 |
|------|-----------|----------|----------|
| 후원 에스크로 결제 모달 | 펀딩 금액 선택, 결제 수단 선택, 에스크로 보관/분할 정산 안내 및 동의 체크를 포함한 결제 폼 | 그대로 사용 | 결제 처리 로딩 스피너 및 성공 화면을 세련되게 구현하여 높은 심미성과 거래 신뢰감을 선사함 |
| 프로젝트 등록 위저드 모달 | 기본 정보(1단계), 예산 및 정산 계획(2단계), 에스크로 서약(3단계) 양식 및 동적 리스트 갱신 | 그대로 사용 | 에스크로 본질에 맞추어 예산 사용 계획과 서약 단계를 필수 기재하도록 흐름을 체계화함 |
| 마이페이지 에스크로 대시보드 | 에스크로 지표 카드, 후원/개설 탭, 후원 상태 배지(`보관 중`, `정산 완료`, `환불 완료`) 제공 | 그대로 사용 | 사용자가 본인의 자금이 어떻게 에스크로 계좌에 묶여 있는지 명확히 인지하고 조작할 수 있음 |
| 실시간 환불 처리 API 및 UI | 대시보드에서 `보관 중`인 후원 건을 즉시 취소 및 환불 조치하는 단일 API 및 UI 트리거 | 그대로 사용 | 취소 완료 즉시 프로젝트의 모금 현황 및 달성률이 차감 연동되어 투명하고 안전한 에스크로 거래를 증명함 |

**소감**
단순 데모나 알림 처리를 넘어, 인메모리 상에서 모금액 합산 및 환불 차감, 프로젝트 등록이 실시간으로 동적 갱신되는 완성도 높은 풀스택 에스크로 프로토타입을 구성함으로써 서비스의 핵심 비즈니스 모델을 구체적으로 시각화할 수 있었습니다.
