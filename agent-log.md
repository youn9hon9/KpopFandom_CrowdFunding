# Agent Log 작성 가이드

> 프로젝트 주차별로 Agent를 어떻게 활용했는지를 기록하기 위한 문서  
> 기능 단위로 Agent 활용 내용을 기록
> 아래 형식을 하나의 단위로, 구현한 기능마다 반복 작성
> 단, 모든 Agent 활용을 기록할 필요는 없음. **의미 있는 상호작용**을 선별해 작성

---

## 작성 원칙

- 프롬프트는 **실제로 사용한 내용**을 그대로 옮기되, 맥락을 함께 기술해야 함.
- Agent의 작업 결과를 그대로 활용한 경우도 정직하게 기록하되, **왜 그대로 활용하였는지** 이유를 남길 것.
- Agent의 작업 결과를 수정한 경우 구체적으로 기록할 것.
- 오류 사례가 없다면 해당 항목은 생략.

---

```
|---|--------------|-------------------------------|----------------|
| 1 | | | |
| 2 | | | |

**생성 결과 vs 실제 반영**

| 항목 | Agent 결과 | 실제 반영 | 수정 이유 |
|------|-----------|----------|----------|
|  |  |  |  |

**오류 사례** *(해당 없으면 생략)*

- **상황**: 어떤 요청을 했는가
- **문제점**: Agent 결과의 무엇이 문제였는가

**소감**
이 기능에서 Agent 활용이 어땠는지 한두 줄 자유롭게.
```

## 구글 및 카카오 소셜 로그인 (OAuth) — 실제 연동 전환

**목적**
Mock 시뮬레이터 기반 Dual-Mode 인증을 제거하고, Google·Kakao **실제 OAuth 2.0 / OpenID Connect** 연동으로 전환함. PKCE·state 검증, 표준 OIDC 클라이언트 라이브러리 도입, `.env` 기반 자격 증명 관리로 프로덕션에 가까운 인증 흐름을 구성함.

**활용 라이브러리**


| 라이브러리                                                         | 버전    | 역할                                                                                          |
| ------------------------------------------------------------- | ----- | ------------------------------------------------------------------------------------------- |
| `[openid-client](https://github.com/panva/openid-client)`     | ^6.x  | Google OIDC discovery, Authorization Code + PKCE, `authorizationCodeGrant`, `fetchUserInfo` |
| `[cookie-parser](https://github.com/expressjs/cookie-parser)` | ^1.x  | Express `req.cookies` 파싱 (수동 split 파서 제거)                                                   |
| `[dotenv](https://github.com/motdotla/dotenv)`                | ^16.x | `.env` 로컬 자격 증명 로드                                                                          |
| Node.js 내장 `fetch` / `crypto`                                 | —     | Kakao 토큰·프로필 API, OAuth state·세션 ID 생성                                                      |


**요청 및 프롬프트**


| #   | 요청 내용 요약              | 실제 사용한 프롬프트 (핵심 부분)                                                     | 효과적이었던 이유                               |
| --- | --------------------- | ----------------------------------------------------------------------- | --------------------------------------- |
| 1   | Mock 제거 및 실제 OAuth 전환 | "현재 소셜 로그인을 진짜로 바꾸고, agent-log.md도 업데이트해줘"                              | Dual-Mode 제거 범위와 문서 동기화를 한 번에 정리할 수 있었음 |
| 2   | 라이브러리 선정 및 PKCE 적용    | "Google은 openid-client로 OIDC discovery + PKCE 쓰고, Kakao는 state 검증 추가해줘" | raw fetch만 쓰던 Google 토큰 교환을 표준 플로우로 교체  |
| 3   | 미설정 시 UX              | "mock-login.html 대신 oauth-setup.html 안내 페이지로 보내줘"                       | 키 없을 때 빈 화면 대신 설정 절차를 안내                |


**생성 결과 vs 실제 반영**


| 항목             | Agent 결과                                        | 실제 반영  | 수정 이유                                                                |
| -------------- | ----------------------------------------------- | ------ | -------------------------------------------------------------------- |
| Google OAuth   | `lib/oauth.js` + `openid-client` discovery/PKCE | 그대로 사용 | v6 API(`buildAuthorizationUrl`, `authorizationCodeGrant`) 문법 확인 후 적용 |
| Kakao OAuth    | fetch 기반 + `state` CSRF 검증                      | 그대로 사용 | Kakao는 OIDC discovery 미지원이라 수동 구현 유지                                 |
| Mock 시뮬레이터     | `mock-login.html` 삭제                            | 그대로 반영 | 실제 OAuth만 허용하는 정책에 맞춤                                                |
| 쿠키 파싱          | `cookie-parser` 미들웨어                            | 그대로 사용 | 수동 파서에서 `=` 포함 값 깨지는 문제 예방                                           |
| OAuth state 저장 | 인메모리 Map + 10분 TTL                              | 그대로 사용 | 서버 재시작 시 state 소실 — 에러 메시지로 안내하도록 보완                                 |


**오류 사례**

- **상황**: Google 로그인 후 `redirect_uri_mismatch` (400) 발생
- **문제점**: Google Cloud Console에 `http://127.0.0.1:3000/auth/google/callback`만 등록했는데, 브라우저는 `http://localhost:3000`으로 접속함. OAuth는 호스트명까지 origin 전체가 일치해야 함.
- **수정 내용**: `.env`의 `REDIRECT_URI_BASE`를 실제 접속 URL과 동일하게 맞추고, Console에 두 URI 모두 등록.
- **상황**: Kakao 토큰 교환 시 `KOE010: invalid_grant` / Client Secret 관련 오류
- **문제점**: Kakao Developers에서 Client Secret을 **활성화**해 두었는데 `.env`에는 `KAKAO_CLIENT_SECRET`을 비워 둠. 반대로 Secret 비활성화 상태에서 값을 넣어도 실패.
- **수정 내용**: Kakao 콘솔 보안 설정과 `.env` 값을 1:1로 맞춤. REST API 키(JavaScript 키 아님)를 `KAKAO_CLIENT_ID`에 사용.
- **상황**: Google 콜백에서 `OAuth state가 만료되었거나 일치하지 않습니다` — 로그인 직후에도 실패
- **문제점**: `nodemon`/`npm start` 재시작으로 인메모리 `oauthStates` Map이 초기화됐는데, 브라우저는 이전 탭의 Google redirect를 그대로 따라옴.
- **수정 내용**: state missmatch 시 사용자 친화적 메시지 + 메인으로 redirect. 개발 중에는 로그인 **시작부터** 다시 시도하도록 문서화.
- **상황**: `openid-client` v5 예제 코드를 v6에 그대로 붙여넣어 `Issuer.discover is not a function` 발생
- **문제점**: v6에서 API surface가 `discovery()`, `buildAuthorizationUrl()`, `authorizationCodeGrant()`로 변경됨.
- **수정 내용**: README Quick start와 `build/index.d.ts` 타입 정의를 대조해 v6 문법으로 전면 수정.
- **상황**: 로그인 성공했는데 헤더 아바타가 갱신되지 않음 — "CORS 문제인가?" 하고 30분 헤매임
- **문제점**: OAuth redirect는 same-origin이라 CORS와 무관. `Set-Cookie`는 됐지만 상세 페이지에서 `auth.js`보다 `detail.js`가 먼저 렌더링되며 race condition처럼 보였음.
- **수정 내용**: `checkLoginStatus()`를 `DOMContentLoaded`에서 OAuth 에러 처리 후 호출하도록 순서 정리. 쿠키는 DevTools Application 탭에서 확인.
- **상황**: Kakao 로그인 후 이메일이 `@kakao.user` placeholder로만 표시
- **문제점**: Kakao **카카오계정(이메일)** 동의 항목 미체크 + 비즈 앱 심사 전에는 `kakao_account.email`이 내려오지 않음.
- **수정 내용**: 동의 항목 설정 가이드를 `.env.example`에 추가. 이메일 없을 때 `${userData.id}@kakao.user` fallback 유지.

**소감**
Mock 없이 실제 OAuth만 붙이니 `redirect_uri` 한 글자, REST API 키 vs JavaScript 키, Client Secret on/off 같은 "콘솔 설정 지뢰"를 전부 밟아 봤다. `openid-client` v6 API 변경과 인메모리 state 특성까지 겹치면서 트러블슈팅 시간이 길었지만, PKCE·state 검증이 들어간 뒤에는 보안적으로 훨씬 납득 가는 구조가 됐다.

## 안전 에스크로 결제 프로세스 (후원, 서비스 등록, 환불)

**목적**
`docs/모두의 창업...md`에 명시된 비즈니스 목표에 따라 자금 횡령 및 무산 리스크를 차단할 수 있는 안전 에스크로 펀딩 거래, 정산 계획 수립을 포함한 프로젝트 올리기 양식, 그리고 사용자가 직접 거래 상황을 모니터링하고 환불할 수 있는 마이페이지 에스크로 대시보드를 구축함.

**요청 및 프롬프트**


| #   | 요청 내용 요약                       | 실제 사용한 프롬프트 (핵심 부분)                                                                             | 효과적이었던 이유                                                                                                |
| --- | ------------------------------ | ----------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| 1   | 에스크로 결제 화면 프로세스 및 환불/서비스 등록 구현 | "환불이나 서비스 등록, 후원하기 등 결제와 관련된 에스크로 결제 화면 프로세스를 구현하고, 에이전트 로그에 추가해줘. 자세한 프로세스는 docs 내의 문서들을 확인해." | `docs` 내 비즈니스 요건서(먹튀 방지, 100% 자동 환불 약정, 단계별 정산)를 정확히 파악해 실시간 연동이 되는 백엔드 API와 다단계 모달 UI를 일관성 있게 유기적으로 작성함 |
| 2   | 마이페이지에 에스크로 대시보드 추가            | "마이페이지에 내가 후원한 프로젝트들의 에스크로 결제 상태를 보여주고 환불할 수 있게 해줘"                                             |                                                                                                          |
| 3   | 프로젝트 개설 프로세스에 에스크로 결제 프로세스 추가  | "프로젝트 개설 프로세스에 에스크로 결제 프로세스를 추가해줘"                                                              |                                                                                                          |


**생성 결과 vs 실제 반영**


| 항목                 | Agent 결과                                                   | 실제 반영  | 수정 이유                                                     |
| ------------------ | ---------------------------------------------------------- | ------ | --------------------------------------------------------- |
| 후원 에스크로 결제 모달      | 펀딩 금액 선택, 결제 수단 선택, 에스크로 보관/분할 정산 안내 및 동의 체크를 포함한 결제 폼     | 그대로 사용 | 결제 처리 로딩 스피너 및 성공 화면을 세련되게 구현하여 높은 심미성과 거래 신뢰감을 선사함       |
| 프로젝트 등록 위저드 모달     | 기본 정보(1단계), 예산 및 정산 계획(2단계), 에스크로 서약(3단계) 양식 및 동적 리스트 갱신   | 그대로 사용 | 에스크로 본질에 맞추어 예산 사용 계획과 서약 단계를 필수 기재하도록 흐름을 체계화함           |
| 마이페이지 에스크로 대시보드    | 에스크로 지표 카드, 후원/개설 탭, 후원 상태 배지(`보관 중`, `정산 완료`, `환불 완료`) 제공 | 그대로 사용 | 사용자가 본인의 자금이 어떻게 에스크로 계좌에 묶여 있는지 명확히 인지하고 조작할 수 있음        |
| 실시간 환불 처리 API 및 UI | 대시보드에서 `보관 중`인 후원 건을 즉시 취소 및 환불 조치하는 단일 API 및 UI 트리거       | 그대로 사용 | 취소 완료 즉시 프로젝트의 모금 현황 및 달성률이 차감 연동되어 투명하고 안전한 에스크로 거래를 증명함 |


**소감**
단순 데모나 알림 처리를 넘어, 인메모리 상에서 모금액 합산 및 환불 차감, 프로젝트 등록이 실시간으로 동적 갱신되는 완성도 높은 풀스택 에스크로 프로토타입을 구축함으로써 서비스의 핵심 비즈니스 모델을 구체적으로 시각화할 수 있었습니다.

## HTTP 에러 코드 및 상황별 UX (404 / 401 / 500)

**목적**
심사 기준「상황에 맞는 에러 코드 및 UX 제공」을 충족하기 위해, API JSON 응답에 HTTP 상태 코드·`code` 필드를 통일하고, 전용 에러 안내 화면·토스트 알림·401 시 로그인 모달 자동 유도를 구현함.

**활용 라이브러리 / 패턴**


| 구성                          | 역할                                                      |
| --------------------------- | ------------------------------------------------------- |
| Express 4 내장 `res.status()` | API·페이지별 HTTP 상태 코드 반환                                  |
| `lib/asyncHandler.js`       | async 라우트 미처리 reject → 전역 500 핸들러로 전달                   |
| `public/errors.js`          | `handleApiResponse`, `showToast`, `parseApiError` 공통 UX |
| `public/error.html`         | 401·404·500 통합 안내 화면 (`?code=` 쿼리)                      |


**요청 및 프롬프트**


| #   | 요청 내용 요약 | 실제 사용한 프롬프트 (핵심 부분)                                | 효과적이었던 이유                |
| --- | -------- | -------------------------------------------------- | ------------------------ |
| 1   | 에러 UX 구현 | "404, 401, 500 등 상황별 안내 화면 처리 구현하고 agent-log에도 추가" | API·페이지·프론트 3계층을 한 번에 정리 |
| 2   | alert 제거 | "401이면 로그인 모달, 500은 전용 화면, alert 대신 토스트"           | UX 일관성과 심사 요건 동시 충족      |


**생성 결과 vs 실제 반영**


| 항목           | Agent 결과                                                | 실제 반영  | 수정 이유                               |
| ------------ | ------------------------------------------------------- | ------ | ----------------------------------- |
| 전역 404 (페이지) | 존재하지 않는 URL → `/error.html?code=404`                    | 그대로 사용 | Express static 이후 fallback으로 처리     |
| API 404      | `app.use('/api', ...)` JSON `{ code: 'API_NOT_FOUND' }` | 그대로 사용 | HTML redirect와 API JSON 분리          |
| 전역 500       | 4-인자 error middleware                                   | 그대로 사용 | API는 JSON, 페이지는 error.html redirect |
| 프론트 공통       | `errors.js` + 토스트 UI                                    | 그대로 사용 | `auth.js` 후원·등록·환불·마이페이지에 적용        |
| 상세 404/500   | `renderNotFound` / `renderServerError` 분리               | 그대로 사용 | 404와 500을 같은 UI로 처리하지 않음            |


**오류 사례**

- **상황**: 404 fallback을 `express.static` **앞**에 두어 CSS·JS·이미지까지 전부 error.html로 redirect됨
- **문제점**: 미들웨어 순서 오류. 정적 파일 요청(`/style.css`)도 catch-all에 걸려 스타일이 깨진 채 에러 페이지만 표시됨.
- **수정 내용**: `express.static` → API 라우트 → `/api` 404 → 페이지 404 순서로 재배치.
- **상황**: API 401인데 프론트에서 `alert('로그인이 필요합니다')`만 띄우고 끝 — 사용자가 로그인 버튼을 다시 찾아야 함
- **문제점**: 401 UX가「알림」에 그치고「복구 동작」이 없음.
- **수정 내용**: `handleApiResponse`에서 401 감지 시 `showLoginModal()` 자동 호출 + warning 토스트.
- **상황**: `handleApiResponse`에서 `res.json()`을 두 번 호출해 `body stream already read` TypeError
- **문제점**: `!res.ok` 분기에서 `parseApiError(res)`로 body를 consume한 뒤, 호출부에서 또 `res.json()` 시도.
- **수정 내용**: `handleApiResponse`가 `{ ok, data, status, message }` 객체를 반환하도록 통일하고, 호출부는 `result.data`만 사용.
- **상황**: `/error.html?code=404` 접속 시 무한 redirect
- **문제점**: catch-all이 `error.html` 요청도 404로 간주해 `?from=/error.html` 붙여 재redirect.
- **수정 내용**: `express.static`이 `error.html`을 먼저 서빙하므로 catch-all에 도달하지 않음 — 순서 확인으로 해결.
- **상황**: async OAuth 콜백에서 throw된 에러가 `[object Object]`만 찍히고 클라이언트는 빈 화면
- **문제점**: Express 4는 async 함수 reject를 자동 catch하지 않음.
- **수정 내용**: `lib/asyncHandler.js` 도입 + 전역 error middleware에서 `err.stack` 로깅.
- **상황**: 상세 페이지에서 API 500과 404를 동일한 `renderNotFound()`로 처리
- **문제점**:「없는 프로젝트」와「서버 장애」를 사용자가 구분할 수 없음.
- **수정 내용**: `detail.js`에서 status별 분기 — 404는 not-found, 500+는 `renderServerError` + 재시도 버튼.

**소감**
에러 처리는 기능 추가보다「미들웨어 순서」와「response body 한 번만 읽기」 같은 사소한 함정이 반복됐다. alert를 토스트+모달로 바꾸니 401 흐름이 훨씬 자연스러워졌고, `error.html` 하나로 404/401/500을 커버하니 유지보수도 수월했다.

- **상황**: `.env`에 `ADMIN_EMAILS` 넣었는데도 관리자 메뉴가 안 보임
- **문제점**: OAuth 로그인 **이전**에 만든 세션 쿠키가 남아 `role` 없는 프로필 유지. 또는 이메일 대소문자·공백 불일치.
- **수정 내용**: 로그아웃 후 재로그인. `assignRole`에서 `toLowerCase()` 정규화. `ADMIN_EMAILS` 콤마 구분 문서화.
- **상황**: 총대인데 상세 화면이 후원자 뷰로 보임
- **문제점**: 샘플 프로젝트 10개는 `hostUserId: null` — OAuth로 새로 등록한 프로젝트만 총대 매칭됨.
- **수정 내용**: `POST /api/projects` 시 `hostUserId: req.user.id` 저장. 샘플은 의도적으로 미연결(데모 데이터).
- **상황**: 403인데 로그인 모달이 뜸
- **문제점**: `handleApiResponse`가 401·403 모두 `showLoginModal()` 호출.
- **수정 내용**: 403은 토스트만, 401만 로그인 모달 — 권한 없음과 미인증 구분.
- **상황**: `GET /api/projects/:id` 응답 구조 변경 후 상세 페이지 전체 깨짐
- **문제점**: 기존 `res.json(project)` → `{ viewerRole, project, myDonation }` 로 변경됐는데 `renderDetail(project)` 그대로 호출.
- **수정 내용**: `detail.js` 전면 수정, `payload.project` 사용.
- **상황**: `checkLoginStatus`가 `detail.js`가 그린 총대 CTA를 덮어써서 후원 버튼이 다시 나타남
- **문제점**: auth.js가 `.detail-cta .btn-primary`를 무조건 후원 모달에 연결.
- **수정 내용**: `window.detailContext` 도입, `viewerRole === 'backer'`일 때만 `setupDetailSponsorButton` 호출.
- **상황**: 관리자가 `admin.html` 접속 시 404
- **문제점**: catch-all 404가 `admin.html` 정적 파일보다 먼저 실행된다고 착각 — 실제는 `express.static` 순서 문제 아님, **로그인 role 미설정**이었음.
- **수정 내용**: `admin.js`에서 `/api/auth/me` role 확인 후 403 안내 화면 표시.

**소감**
역할을 나누니「한 `detail.html`에 if문 3개」보다 API에서 `viewerRole`을 내려주는 편이 훨씬 낫다는 걸 다시 느꼈다. `hostName`에서 `hostUserId`로 바꾸면서 샘플 데이터와 실사용자 데이터가 섞이는 혼란이 있었지만, 데모와 프로덕션 경계를 README에 명시하면서 정리됐다.

- **상황**: 프로젝트 `community`·`reviewNotes` 등 중첩 객체를 테이블 컬럼으로 정규화하려 함
- **문제점**: 기존 `roles.toPublicProject`·API 응답 형태와 1:1 매핑 비용이 큼.
- **수정 내용**: 프로젝트는 JSON 단일 컬럼(`data`)으로 저장, 후원만 정규화 — 데모 규모에서 마이그레이션 리스크 최소화.
- **상황**: OAuth `state` Map은 여전히 인메모리
- **문제점**: 재시작 직후 진행 중인 OAuth 콜백만 실패할 수 있음 (세션·CRUD와 별개).
- **수정 내용**: 기존 동작 유지. 로그인 완료 후 세션은 DB에 저장되므로 재시작 후에도 로그인 상태 유지.

**소감**
인메모리에서 SQLite 파일 하나로 바꾸는 것만으로도「데모인데 데이터가 날아간다」는 체감이 크게 줄어든다. 프로젝트 JSON 컬럼 방식은 스키마 변경 없이 기존 객체 구조를 그대로 살릴 수 있어서, MongoDB 전환 전 단계로 적당했다.

**목적**
상세 페이지 커뮤니티 탭을 클라이언트 전용 UI에서 **서버 CRUD + SQLite 영속화**로 전환함. 공지·일정·투표·게시글·댓글·좋아요·삭제·내 활동을 한 화면에서 제공.

| 구분          | 엔드포인트                             | 권한          |
| ----------- | --------------------------------- | ----------- |

| 투표          | `POST .../polls/:id/vote`         | 로그인 (1인 1표) |

| 삭제          | `DELETE ...`                      | 작성자·총대·관리자  |
| 내 활동        | `GET /api/users/me/community`     | 로그인         |


**오류 사례**

- **상황**: 기존 샘플 투표에 `id`·`voters` 없음
- **문제점**: 서버 투표 API가 pollId로 찾지 못함.
- **수정 내용**: `ensureCommunity()`에서 로드 시 자동 id·voters 부여, GET 상세 시 DB 저장.
- **상황**: 투표 UI가 클라이언트에서만 결과 표시
- **문제점**: 새로고침 시 투표 결과 소실.
- **수정 내용**: `poll.voters[userId]` + `options[idx].votes` 서버 집계, 재시작 후에도 유지.
- **상황**: `style.css` 전체를 community 스타일로 덮어씀
- **문제점**: 기존 전역 스타일 전부 삭제.
- **수정 내용**: `git checkout -- public/style.css` 복구 후 파일 **끝에** community 블록 append.

**소감**
community를 project JSON 하위로 두니 SQLite 마이그레이션 없이 바로 CRUD를 붙일 수 있었다. 서브탭 UI로「전체 / 내 활동」을 나누니 데모에서도「내가 쓴 글」 확인 흐름이 자연스럽다.
